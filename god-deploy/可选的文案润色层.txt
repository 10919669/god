

这个产品不该把“大模型”当成出结果引擎，而应该把它降成**可选的文案润色层**。真正的出结果能力，应该来自一套**可规则执行、可复现、可降级运行**的算法模拟引擎。你给的规则文件本身也已经把这条路铺好了：后台必须按固定顺序完成当前问事时刻的计算，前台只展示“结论、理由、风险、建议、趋势”，不暴露术语。

---

## 你现在应该把系统拆成 3 层

### 1）规则计算层

这层不依赖大模型，负责根据**当前时间 + 地点 + 问题文本**跑出结构化结果。
你现有材料已经明确了这条主链路：

* 问事时间识别
* 真太阳时校正
* 节气定位
* 月将确定
* 占时确定
* 天地盘
* 四课
* 三传按九宗门发用
* 课体判名
* 神煞与占类映射分析

而且规则文档明确要求：如果这些关键步骤没完成，就不能硬给断语。

### 2）决策翻译层

这层把规则计算出的结构化标签，翻译成前台可用字段，比如：

* 方向：偏 A / 偏 B / 偏等等
* 风险等级：低 / 中 / 高
* 建议动作：立刻做 / 轻试探 / 再观察 / 不宜推进
* 趋势：短期利动 / 短期反复 / 适合求助 / 适合升级

这层也**不需要大模型**，完全可以用模板和规则映射实现。前台只需要看到“你算出了什么、为什么这么判断、风险点、该怎么做、哪些不能做、趋势如何演变”。

### 3）表达增强层

这层才是大模型可选项。
用途不是“算”，而是：

* 把结构化结果润色成更像“小闺蜜”或“宇宙裁决者”的语气
* 做更多文案变体
* 做更自然的分享文案

也就是说，**没大模型时照样能出结果；有大模型时只是更会说话。**

* 月将按节气中的**中气换将**规则确定。

**把“现在”变成后台真正可算的时空上下文。**

这段的目标是：

# 第三段：决策评分与前台翻译

这是最关键的一层，因为它决定你能不能**不用大模型也出完整结果**。

## 3.1 先输出结构化决策字段

我建议先固定成这 8 个字段：

* `decision_bias`：A / B / wait / retry
* `confidence_level`：0-100
* `risk_level`：low / medium / high
* `action_mode`：act_now / test_lightly / wait_and_watch / stop
* `trend_mode`：smooth / fluctuating / delayed / blocked
* `tone_mode`：bestie / oracle
* `share_card_type`：conclusion / versus / ask_help
* `escalation_recommended`：true / false

## 3.2 再做规则打分

一个实用的 MVP 算法可以这样设计：

### 决策总分

`total_score = favorable_score - unfavorable_score + timing_bonus - caution_penalty`

其中：

* `favorable_score`：有利征象累计分
* `unfavorable_score`：不利征象累计分
* `timing_bonus`：当前时点是否利推进
* `caution_penalty`：若是重题且征象混杂，增加谨慎惩罚

### 决策方向

* `total_score >= 3`：偏 A / 偏推进
* `0 <= total_score < 3`：轻偏 A，但建议试探
* `-2 <= total_score < 0`：暂缓，先观察
* `total_score < -2`：不宜推进 / 偏 B / 偏停止

### 风险等级

* 有明显凶征且为重题：high
* 吉凶混杂：medium
* 轻题且结构稳定：low

### 升级触发

满足任一项即 `escalation_recommended = true`：

* 问题语义涉及长期后果
* 涉及关系推进 / 切断
* 结构征象反复
* 决策方向不明但风险高

这就能把后台结果稳定翻成前台产品逻辑，而不用依赖 LLM 即时发挥。

* 理由
* 风险
* 建议

### 轻题模板

输入：

* decision_bias=A
* confidence=72
* risk=low
* tone=bestie

输出：

* 结论：今天更适合选 A
* 理由：你现在更适合轻一点、顺一点的选择
* 建议：先选这个，别再来回刷
* 趋势：今天适合快决策，不适合反复比较

### 重题模板

输入：

* decision_bias=A
* confidence=54
* risk=high
* tone=oracle
* escalation=true

输出：

* 结论：如果非要先走一步，今天更偏 A
* 理由：你真正卡住的不是要不要做，而是怕做了以后收不回来
* 风险：这题最怕仓促，不怕慢半步
* 建议：先小范围试探，不要一次性压满
* 趋势：短期内还会反复，这题值得认真看
* CTA：想看得更深，进入完整推演

这层完全可以不用大模型，靠模板拼接就能产出合格结果。

# 《规则引擎与模板文案映射表》V1

## 1. 文档目标

用于研发实现一个**可规则化、可复现、可降级运行**的结果引擎，支持：

* 用户输入模板题或自由问题
* 系统自动读取当前时间
* 有地点时走精准模式
* 无地点时走快速模式
* 输出前台可见结果，不依赖大模型生成核心判断
* 仍然保持“高频轻决策 → 分享 → 更认真场景 → 导流大宗师”的产品闭环

---

## 2. 总体架构

### 2.1 三层架构

#### A. 上下文计算层

负责把“现在”变成可计算上下文。

输入：

* 当前时间
* 时区

* 用户问题

输出：

* 修正后时间
* 精度模式
* 占时

* 事件类别

负责把后台结构转成结构化决策字段。

输出：

* 偏向 A / B / 等等
* 风险等级
* 动作建议
* 趋势
* 是否建议升级

#### C. 模板翻译层

负责把结构化字段变成前台人话。

输出：

* 结论
* 理由
* 风险
* 建议
* 趋势
* CTA

这套拆法也符合 AI 执行版对“模块拆分、输入输出、规则、异常、状态”的要求。

# 5. 表三：决策评分表

这张表是“不用大模型也能出结果”的核心。

## 5.1 评分主公式

```text
total_score
= favorable_count * 2
- unfavorable_count * 2
+ movement_signal
+ support_signal
- caution_signal
- delay_signal
- conflict_signal
- void_signal
+ precision_bonus
- boundary_penalty
```

## 5.2 精度系数

| precision_mode | precision_bonus | 说明       |
| -------------- | --------------: | -------- |
| precise        |              +1 | 有地点，已校正  |
| manual_city    |               0 | 手动城市，半精准 |
| fast           |              -1 | 无地点，快速模式 |

## 5.3 边界惩罚

| 情况       | boundary_penalty |
| -------- | ---------------: |
| 靠近子时边界   |                1 |
| 靠近中气切换   |                1 |
| 同时靠近两个边界 |                2 |
| 无边界风险    |                0 |

这一步是必须的，因为你的底层文件明确要求：接近子时、节气尤其是中气切换时，不能粗略处理；地点缺失也不能伪装成已校正状态。

## 5.4 方向判定

| total_score 区间 | decision_bias | 解释          |
| -------------- | ------------- | ----------- |
| >= 4           | A             | 明显偏 A / 宜推进 |
| 1 ~ 3          | lean_A        | 轻偏 A / 可试探  |
| -1 ~ 0         | wait          | 暂缓 / 先观察    |
| -4 ~ -2        | lean_B        | 轻偏 B / 不宜贸然 |
| <= -5          | B             | 明显偏 B / 宜收手 |

## 5.5 风险等级

| 条件                                      | risk_level |
| --------------------------------------- | ---------- |
| L1 且 total_score >= 1 且 conflict/void 低 | low        |
| 吉凶混杂，或 L2                               | medium     |
| L3 且 caution/conflict/delay 高           | high       |

## 5.6 动作模式

| decision_bias | risk_level  | action_mode      |
| ------------- | ----------- | ---------------- |
| A             | low         | act_now          |
| lean_A        | medium/high | test_lightly     |
| wait          | 任意          | wait_and_watch   |
| lean_B        | medium      | do_not_rush      |
| B             | high        | stop_or_escalate |

## 5.7 趋势模式

| 结构特征                 | trend_mode  |
| -------------------- | ----------- |
| stable + favorable 高 | smooth      |
| mixed + delay 中等     | fluctuating |
| blocked / void 高     | blocked     |
| repeated / reversed  | delayed     |

## 5.8 升级触发规则

满足任一项，`escalation_recommended = true`：

* seriousness_level = L3
* risk_level = high
* decision_bias = wait 且用户问题是关系 / 职业 / 合作
* trend_mode = blocked 或 delayed
* precision_mode = fast 且用户问的是重大问题

这和前面的产品主线一致：轻问题负责拉新和传播，重问题负责自然导流，而不是继续当玩具处理。

## 6.2 语气模式

| tone_mode |    适用层级 | 风格           |
| --------- | ------: | ------------ |
| bestie    | L1 / L2 | 懂你的小闺蜜       |
| oracle    |      L3 | 带一点神谕感的宇宙裁决者 |

## 6.3 轻题模板映射

### 模板 A：L1 + A / lean_A

**title**

* 今天更适合选 A
* 今天更偏这个
* 这一题别绕了，就它

**reason**

* 你现在更需要顺一点的选择
* 今天更适合轻一点、快一点
* 这件事拖着不会更好，只会更烦

**action**

* 先选这个，别继续来回刷
* 就按这个走，先把决定做掉
* 这一题不值得再比三轮

**trend**

* 今天适合快决策，不适合反复比较
* 越拖越容易失去当下感觉

### 模板 B：L1 + wait / lean_B

**title**

* 这题今天别急着定
* 现在还不是最顺的时候
* 这一题先缓一下更稳

**reason**

* 你现在不是没答案，是心还没定
* 今天更像容易冲动，不像适合拍板

**action**

* 先放一会儿，晚点再看
* 别被一时情绪推着选

**trend**

* 短时间内会反复，硬定不划算

## 6.4 社交题模板映射

### L2 + A / lean_A

**title**

* 今天宇宙站 A
* 这局更偏 A
* 真要选边，我站 A

**reason**

* A 更贴你现在的状态
* A 今天更有气势 / 更顺手

**action**

* 发给朋友，看看他们站哪边
* 先按 A 想，再听听外部意见

**trend**

* 这是适合互动求认同的一题

### L2 + wait

**title**

* 这局别急着站队
* 今天更适合先看，不适合马上押注

**reason**

* 两边现在都不够干脆
* 你卡住的点还没真正被说出来

**action**

* 先丢给朋友看看他们怎么选
* 先问一圈，再做最后决定

## 6.5 重题模板映射

### L3 + A / lean_A + high risk

**title**

* 如果非要先走一步，今天更偏 A
* 这题不是毫无机会，但更适合小步推进

**reason**

* 你真正纠结的不是要不要做，而是怕做了以后收不回来
* 现在不是不能动，是不能一下走太满

**risk**

* 这题更怕仓促，不怕慢半步
* 风险不在第一步，在后续承接

**action**

* 先试探，不要一次性压满
* 先做低成本动作，再看反馈

**trend**

* 短期还会拉扯，但不是完全不能推进
* 这件事适合先动一点，不适合立刻定局

**cta**

* 先发给你信任的人看看
* 想认真看，再进入完整推演

### L3 + wait / B

**title**

* 这题现在不适合硬推
* 如果要稳，这一步先别走满

**reason**

* 你眼下看到的是决定本身，没看到后面的代价
* 现在更像不稳，不像水到渠成

**risk**

* 真正要防的不是慢，是误判
* 现在强推进，后面补救成本更高

**action**

* 先停半步，整理条件再决定
* 不要为了结束焦虑而仓促做结论

**trend**

* 短期仍有反复
* 这题值得换到更深层去看

**cta**

* 先别急着定局
* 进入完整推演，看清后续影响

# 9. MVP 先做什么

按照你前面的产品主线，现在最该优先的是最小闭环，不是一次做全。

* 当前时间采集
* 精准 / 快速模式
* L1 / L3 分层
* 决策评分
* 模板文案引擎

## P1 再做

* L2 社交题精细化

## 暂不做

* 强个性化文风

# 10. 给研发的验收标准

### 验收 1：无大模型也能稳定出结果

* 给定用户输入一个有效问题
* 当系统获取到当前时间并完成上下文准备
* 那么系统必须返回结构化结果 JSON
* 并且至少包含 `decision_bias / risk_level / action_mode / trend_mode`

### 验收 2：无地点时能降级而不是报废

* 给定用户拒绝定位
* 当用户继续提交问题
* 那么系统进入 `fast` 模式
* 并且仍返回结果
* 并且结果中带有低精度标记

### 验收 3：重题必须触发更谨慎模板

* 给定问题被识别为 L3
* 当结果返回
* 那么前台必须使用 `oracle` 语气模板
* 并且展示 `risk` 与 `cta`

### 验收 4：前台不得泄露后台术语

* 给定任意结果页
* 当用户查看内容
* 那么页面文案中不得出现底层推演术语
* 并且只显示结论、理由、风险、建议、趋势

这类需求写法也符合“Given / When / Then”的执行化规范。
